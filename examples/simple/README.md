# examples/simple — Using entcrypt with the entc extension (auto-register)

This example shows how to use `entcrypt` with the **built-in codegen extension**
for **automatic field registration**. The extension scans your ent schemas at
codegen time and auto-discovers encrypted fields, so you don't have to
manually register them.

## How it works

1. **Schema** (`ent/schema/user.go`) — Fields are annotated with `entcrypt.EncryptedField{}`.
2. **Codegen** (`cmd/entc/main.go`) — A custom entc command that passes
   `entcrypt.Extension{}` to the ent codegen pipeline.
3. **Registry** (`ent/entcrypt_gen.go`) — **Auto-generated**. The extension
   writes this file with calls to `entcrypt.Register()` during codegen.
4. **Runtime** (`main.go`) — Hooks and interceptors handle encryption/decryption
   transparently.

## Why use this approach?

- You want encrypted fields to be **auto-registered** during codegen.
- No need to manually maintain a list of encrypted fields in sync with your schema.
- Single source of truth: the `EncryptedField{}` annotations in your schema.

## Key files to look at

| File | What it does |
|------|-------------|
| `ent/schema/user.go` | Schema defining which fields are encrypted |
| `cmd/entc/main.go` | Custom entc command using `entcrypt.Extension{}` |
| `ent/generate.go` | `go:generate` directive invoking the custom entc command |
| `ent/entcrypt_gen.go` | **Auto-generated** — registers encrypted fields via `entcrypt.Register()` |
| `tools.go` | Pins the `entgo.io/ent/cmd/entc` module so `go mod tidy` keeps it available for codegen |
| `main.go` | Application wiring hooks, interceptors, and queries |

## Run it

```bash
go generate ./ent          # run custom entc with entcrypt.Extension
go run .                   # run the example
```

## What to expect

```
created: id=1 name=Alice email=alice@example.com ssn=000-00-0000
queried: id=1 name=Alice email=alice@example.com ssn=000-00-0000
raw (encrypted): email="v1:AES-256-GCM:..." ssn="v1:AES-256-GCM:..."
```

Same runtime behaviour as the `noentc` example. The key difference is that
this example uses the `entcrypt.Extension{}` to automatically register
encrypted fields instead of maintaining that list manually.

## Compare with `noentc`

| | `simple` | `noentc` |
|---|----------|---------|
| **Codegen** | Custom `cmd/entc` with extension | Standard `go generate ./ent` |
| **Registry** | Auto-generated by extension | Manually maintained |
| **Complexity** | Minimal setup | More control/boilerplate |
| **Sync** | Fields auto-discovered | Must update registration manually |
